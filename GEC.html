About:

Project Details

The Retro Adventure Engine is a simple 2D engine designed to help create Retro-esc Top-Down RPGS, good examples of these types of games being Zelda and Pokemon. The system is programmed in the C++ Language using two libraries, SFML and ImGUI. As of current the System supports rendering sprites from a spritesheet, the creation of character game entities and movement.

Targets:
| Retro

The main appeal of the engine will be its retro theme-ing as such limiations will be forced upon users as part of the engines asthetic and design, an example of this will be sprite size capping at 64x64 and a limited colour depth, with a potential lack of transparency allowed (pngs), instead a alpha colour being chosen by the user. Furthermore, the audio will be compressed to fit the technology of the time.

| LightWeight

The size and scope of the engine is being carefully considered throughout all new editions to be as resource-effecient. An example of this would be the use of C++'s built in short data type having mostly replaced integers in the engine.

| User-Friendly

The engine should be easy to expand upon for the benefit of the users attempting to create a project from the engine. Thusly the core concepts of the engine are being kept simple with classes being made to be as concise as possible. The engine should also be generous to its users in that it should give them detailed error messages when an error occurs and effective error handling will be built in. In addition to this, serious consideration is being given to the structured doccumentation that will accompany the engine.

| Effecient

The core underlying code that the user works from should be optimised, clean and quick.

Highlights:

A little about the creation of this project

Movement:

After Successfully building the graphics system and creating animation sprite functionality I moved on to implimenting it into an actual mechanic. I chose Movement as it was simple and would let me Test the system freely.

I learnt how to use keyboard and mouse inputs within C++ From this project, learning input handling will be essential for keymapping later in the project.

In Later Editions, I hope to create my own vector class that is smaller and more lightweight than the one provided in SFML, I would like to do this to have a more tailored vector class to use for movement.

Animation Selection:

After getting graphics to render on the screen the next step was to get an animation playing. I added animation functionality to my now black-boxed graphics class and now game entities can swap their animations / sprite-sheet texture corrdinates when needed.

I learnt how to create animations for this by moving the partian of the current sprite to the next frame of the animation in the spritesheet. Whilst its not the only way to do animation it is one of the simplist and most effective.

In later editions I hope to find a way to optimise my use of spritesheets as to save on expensive functions that SFML has to call when loading a new image as a sprite.

Rendering:

At the beginning of the project, I determined that establishing a graphics system would be the most logical first step, allowing me to visualize elements on the screen as I developed the Retro Adventure Engine.

Through this process, I learned to create a singleton class in C++, which has proven to be highly beneficial. My graphics class is straightforward and user-friendly, offering a wide range of drawing functions with varying parameters to accommodate different use cases. Additionally, I gained experience in error handling related to texture loading.

The current iteration of the graphics system is lightweight and effective. In future versions of the engine, I aim to implement a feature that allows for the selection of an alpha color, enabling the program to render all pixels of that color as transparent.
